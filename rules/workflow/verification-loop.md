# Verification Loop Rules

没有真实命令输出，就不能声称完成、通过或修复。

## Required Evidence

- 实际运行命令、退出码、关键输出。
- 失败项、未运行项及原因、剩余风险。
- 新流程 change 还必须有 CHANGE_METRICS；旧 change 缺失时说明不适用。

注意：不再要求 self-review 记录（`10_自查报告.md` 已废弃）。

## Depth

- minimal：目标测试或最小复现命令（test-planner 覆盖率核对可选）。
- standard：覆盖率核对 + 构建 + 测试 + lint/typecheck。
- strict：standard + 安全、数据库、回滚相关检查。

## Forbidden Claims

- 应该可以通过、看起来没问题、之前跑过、只改一点不用测。
- 主控 Agent 自己跑测试并声称"通过"（必须由独立子 Agent 执行验证）。

---

# Agent 切换规则

验证失败时，按失败类型和失败次数分级切换执行者。核心原则：同组件连续失败时必须切换 agent，防止同一 agent 在盲点中循环。

## 触发条件

test-planner（验证执行模式）输出的 `09_验证结果.md` 中存在 P0 或 P1 失败，或评估结论为"不充分"。

## 分级切换

### L1 —— 构建/编译/测试基础设施错误

**失败表现**：编译不通过、依赖缺失、测试框架配置错误、测试代码语法错误（非断言逻辑错误）。

**执行者**：主控 Agent 调用 `build-error-resolver`（Java 项目调用 `java-build-resolver`）。

**约束**：
- 最小修复，不改业务逻辑代码。
- 不降低测试断言质量，不跳过 lint/checkstyle/SpotBugs。
- 修复后必须重新调用 `test-planner` 验证模式重新执行全量三步（覆盖率核对 + 测试执行 + 评估）。

**禁止**：
- 通过删除测试、关闭 lint、修改安全配置来绕过失败。
- 顺便改业务代码。

### L2 —— 测试断言不通过或覆盖缺口（首次出现）

**失败表现**：测试断言失败（业务逻辑错误）、覆盖率矩阵显示有未实现的测试用例。

**执行者**：主控 Agent。

**约束**：
- 主控 Agent 根据 test-planner 的失败报告修复业务代码或补充测试代码。
- 修复完成后，**主控 Agent 不得自己跑测试验证**。必须重新调用 `test-planner` 验证模式，由其独立重新执行全量三步。
- 如果上次的缺口仍然存在，test-planner 会在评估结论中标注"未修复"。

**禁止**：
- 主控 Agent 自己运行 `mvn test` 或同类命令来判断"修好了没"。
- 修改测试断言使其"通过"而不解决根因。

### L3 —— 同一 Req ID 或同一测试文件连续两次失败

**失败表现**：同一测试文件或同一 Req ID 关联的测试，经过 L2 修复和重新验证后仍然失败。

**执行者**：主控 Agent 调用 `code-reviewer`。**主控 Agent 不得继续自行修复。**

**约束**：
- code-reviewer 诊断根因，输出到 `11_审查报告.md` 的"验证阻塞诊断"节。
- 可能的诊断结论：
  - 实现方向错误 → 建议回到设计阶段重新讨论方案。
  - 测试用例设计不合理 → 建议回到 plan 阶段修改 `08_验证计划.md`。
  - 存在更深层架构问题 → 建议写 `PENDING_DECISIONS.md`。
- 主控 Agent 根据诊断结论决定下一步（回设计 / 回 plan / 写 PENDING_DECISIONS）。

**禁止**：
- 主控 Agent 无视诊断结论继续修同一段代码。
- code-reviewer 直接修改代码。

### L4 —— 连续三次失败

**失败表现**：同一 Req ID 或同一测试文件经过 L3 诊断后仍然失败。

**执行者**：无人。**强制停止。**

**处理**：
- 主控 Agent 写 `PENDING_DECISIONS.md`，详细记录三次失败的经过、每次修复的内容、test-planner 的评估结果。
- 标记 change 为 blocked（`.workflow_state` 中记录）。
- 请求人工介入决策。
- 在人工决策前，不得继续修改代码或运行测试。

## 关键原则

1. **主控 Agent 永远不自己跑测试。** 每一次验证——无论是首次还是修复后——都必须由 test-planner 验证模式独立执行全量三步。
2. **同一组件连续失败必须切换 agent。** L2 → L3 的切换是核心：第二次失败后，诊断者不再是实现者。
3. **不伪造验证。** "我改了代码，应该可以了" → 这不是验证。必须重新跑完整流程。
4. **人工介入是安全阀而非捷径。** L4 是最终防线，触发时应带来有价值的诊断信息。
